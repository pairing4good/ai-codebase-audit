AI Codebase Audit
Automated Security & Quality Analysis

A Docker-based tool that runs parallel Claude agents against your codebase to produce a confidence-ranked top‑10 list of security vulnerabilities, architectural issues, and technical debt — validated by static analysis tools and an adversarial challenge stage.

Supports Java · JavaScript / TypeScript · Python · .NET

The Problem

Static analysis tools are noisy. Manual code reviews are slow and inconsistent. Neither gives you a prioritized, high-confidence list of what actually matters.

Static tools alone

Hundreds of findings, most low-severity. No architectural context. High false-positive rate on real codebases.

Manual review alone

Depends on who reviews it. Misses patterns across files. Doesn't scale to multiple projects.

Single-pass AI

One model, one prompt, one perspective. No cross-validation. No way to measure confidence in findings.

This tool

Multiple independent agents + static tools + convergence scoring + adversarial challenge = high-confidence results.

The 6-Stage Analytical Funnel

Every audit passes through six stages. Each stage narrows and validates the findings from the previous one.

01
Artifact Generation — Map architecture, data flows, entity relationships, and tech debt surface
02
Parallel Agent Analysis — Four isolated specialist agents: architecture, security, maintainability, dependency
03
Static Analysis Tools — Semgrep, SpotBugs, Bandit, ESLint, Snyk, Trivy, and more per language
04
Reconciliation — Merge all sources, calculate convergence scores, flag contradictions
05
Adversarial Challenge — Independent skeptic attacks every finding to eliminate false positives
06
Final Synthesis — Prioritized top-10 report with code snippets, evidence, and fix recommendations

Convergence = Confidence

The core idea: when multiple independent sources flag the same code, you can trust that finding. Agents never see each other's output. Static tools use different detection methods. Agreement is earned, not assumed.

High confidence

2+ agents and 1+ static tool independently identified the same issue at the same location.

Medium confidence

2+ agents or 2+ static tools flagged the issue, but not both categories.

Low confidence

Single source only. Still reported, but ranked lower and subject to adversarial challenge.

Getting Started

Three steps from clone to results.

  • Create a workspace — a directory with your projects alongside the config files copied from the repo
  • Configure — set your API key in .env
  • Configure — list your projects in config.yml
  • Run — docker compose build && docker compose run --rm skills
# config.yml
runner:
  model: claude-sonnet-4-6
  concurrency: 3
  max_budget_usd: 10.0

targets:
  - dir: ofbiz-framework
    skills: [/audit-java]

  - dir: todo-app-flask-reactjs
    skills:
      - /audit-python
      - /audit-javascript

  - dir: WebVella-ERP
    skills: [/audit-dotnet]

Workspace Layout

The workspace directory contains your projects, shared config, and all output.

~/code-audits/ ├── config.yml ← audit targets & settings ├── CLAUDE.md ← agent environment guide ├── .claude/ ← agents, skills, tools │ ├── agents/ │ │ ├── architecture-analyzer.md │ │ ├── security-analyzer.md │ │ ├── maintainability-analyzer.md │ │ ├── dependency-analyzer.md │ │ ├── reconciliation-agent.md │ │ └── adversarial-agent.md │ └── skills/ │ ├── audit-java/ │ ├── audit-javascript/ │ ├── audit-python/ │ └── audit-dotnet/ ├── ofbiz-framework/ ← Java project ├── todo-app-flask-reactjs/ ← Python + JS project ├── WebVella-ERP/ ← .NET project └── logs/ ← all run logs

At container startup, the entrypoint copies .claude/ and CLAUDE.md into each project directory so every agent finds its instructions at the project root.

Results are written to .analysis/<language>/ inside each project — keeping parallel runs isolated with no conflicts.

Execution Model

Each audit task runs inside an ephemeral Docker container with all four language runtimes and pre-installed static analysis tools.

# Build and run from the repo directory
$ docker compose build
$ docker compose run --rm skills

# The orchestrator queues tasks breadth-first:
QUEUED  ofbiz-framework:/audit-java
QUEUED  todo-app-flask-reactjs:/audit-python
QUEUED  WebVella-ERP:/audit-dotnet
QUEUED  todo-app-flask-reactjs:/audit-javascript

# With concurrency: 3, three run simultaneously.
# The fourth starts when a slot opens.
Pre-installed tools

All languages: Semgrep, Snyk, Trivy

Java: SpotBugs, PMD, Checkstyle, OWASP Dep-Check

Python: Bandit, Pylint, mypy, Safety, Radon

JS/TS: ESLint, npm audit

.NET: Roslyn, Security Code Scan, dotnet-outdated

Security model

Ephemeral containers destroyed after each run. Config files mounted read-only. Output restricted to .analysis/ and logs/.

What You Get

Each audit produces a full stage-by-stage trail plus four executive deliverables.

ANALYSIS-REPORT.md

Top 10 findings with code snippets, impact analysis, fix recommendations, and which agents/tools identified each one.

CONFIDENCE-MATRIX.md

Evidence transparency table — a grid showing every finding against every agent and tool, so you can see exactly where confidence comes from.

FINDINGS-DETAILED.json

Complete structured data for all upheld findings. Machine-readable for integration with issue trackers or dashboards.

ARCHITECTURE-OVERVIEW.md

System architecture documentation generated during the artifact stage — component diagrams, data flows, entity relationships.

Plus full stage-by-stage artifacts in .analysis/<language>/stage1-artifacts/ through stage6-final-synthesis/
